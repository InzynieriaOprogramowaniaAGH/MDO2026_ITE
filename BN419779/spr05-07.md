# 1. Jenkins i Blue Ocean
Podstawą współczesnych praktyk DevOps jest całkowita automatyzacja procesów wytwarzania i dostarczania oprogramowania. Wdrożenie serwera Jenkins w architekturze opartej na kontenerach umożliwia zbudowanie środowiska orkiestracji, które wyróżnia się dużą elastycznością, wysoką skalowalnością oraz pełną izolacją zadań.

## Środowisko uruchomieniowe Jenkins i Blue Ocean
Aby zaoferować użytkownikom nowoczesny interfejs oraz zwiększyć czytelność logiki CI/CD, powszechnie wykorzystuje się wtyczkę Blue Ocean. Przekształca ona potoki zadań w graficzną, intuicyjną reprezentację obejmującą etapy sekwencyjne i równoległe. Takie podejście drastycznie skraca czas diagnozowania problemów na ścieżce krytycznej. Cała infrastruktura operuje w obrębie dedykowanej sieci mostkowej Dockera (`bridge network`). Zapewnia to hermetyzację ruchu względem sieci hosta oraz bezpieczną komunikację pomiędzy serwerem głównym a jego agentami.

Standardowy proces tworzenia sieci i powoływania instancji wygląda następująco:
```bash
docker network create jenkins-net

# Uruchomienie głównego węzła Jenkins
docker run --name jenkins-blueocean --restart=on-failure --detach \
  --network jenkins-net --env DOCKER_HOST=tcp://docker:2376 \
  -p 8080:8080 -p 50000:50000 \
  -v jenkins-data:/var/jenkins_home \
  myjenkins-blueocean:latest
```

## Technologia Docker-in-Docker (DinD)
Niezwykle ważnym rozwiązaniem architektonicznym jest nadanie Jenkinsowi możliwości zarządzania kontenerami bez konieczności instalowania samego demona Dockera wewnątrz jego własnego obrazu. Taka instalacja rodziłaby liczne komplikacje związane z bezpieczeństwem i uprawnieniami. Zamiast tego implementuje się wzorzec DinD (Docker-in-Docker). Jenkins komunikuje się tu z zewnętrznym, pomocniczym kontenerem uruchomionym w trybie uprzywilejowanym.

```bash
docker run --name jenkins-docker --rm --detach \
  --privileged --network jenkins-net --network-alias docker \
  --env DOCKER_TLS_CERTDIR=/certs \
  -v jenkins-docker-certs:/certs/client \
  -v jenkins-data:/var/jenkins_home \
  docker:dind
```
W rezultacie operacje budowania obrazów z aplikacjami toczą się w kompletnym odosobnieniu, nie zagrażając stabilności centralnego serwera orkiestracji.

# 2. Pipeline w postaci Jenkinsfile
Dzisiejsze standardy CI/CD opierają się na definiowaniu potoków za pomocą kodu (podejście Infrastructure as Code). Służy do tego plik `Jenkinsfile`, który przechowuje się bezpośrednio w systemie kontroli wersji (np. Git). Zapewnia to pełną historię zmian, ułatwia przeglądy kodu (code review) i pozwala szybko przywrócić poprawną konfigurację w razie awarii.

## Struktura i integracja z SCM
Mechanizm budowania aplikacji stanowi integralną część kodu źródłowego. Dzięki użyciu dyrektywy `checkout scm`, serwer Jenkins potrafi samodzielnie sklonować właściwą gałąź repozytorium zaraz po wykryciu nowych zmian (np. za pośrednictwem webhooków). Struktura potoku dzieli się na czytelne etapy (stages) – zazwyczaj obejmują one przygotowanie środowiska (Setup), kompilację (Build), weryfikację (Test), publikację (Publish) oraz sprzątanie (Cleanup).

Implementacja zmiennych środowiskowych ułatwia z kolei operowanie wersjami i nazewnictwem. Oto rozbudowany wariant deklaratywnego pipeline'u:

```groovy
pipeline {
    agent any
    environment {
        IMAGE_NAME = "my-cpp-app"
        APP_VERSION = "1.0.${env.BUILD_NUMBER}"
    }
    stages {
        stage('Clone SCM') {
            steps { 
                checkout scm 
                sh 'echo "Zainicjowano pobieranie kodu dla wersji: ${APP_VERSION}"'
            }
        }
        stage('Docker Build & Test') {
            steps {
                sh '''
                docker build \
                  --no-cache \
                  --build-arg VERSION=${APP_VERSION} \
                  -t ${IMAGE_NAME}:${APP_VERSION} .
                '''
            }
        }
    }
}
```

# 3. Multi-stage Dockerfile w Jenkins
Pakowanie oprogramowania do kontenerów wymusza na inżynierach rygorystyczne dbanie o bezpieczeństwo i minimalizację wagi ostatecznych obrazów. Odpowiedzią na te wyzwania jest technika Multi-stage Build.

## Separacja środowisk: Builder, Tester, Deploy
W ramach jednego pliku `Dockerfile` projektuje się niezależne warstwy (etapy wydzielone słowem kluczowym `AS`). Pozwala to na radykalne oddzielenie ciężkiego środowiska deweloperskiego od lekkiego, docelowego obrazu produkcyjnego:

1. **Stage Builder:** Posiada bogaty zestaw kompilatorów i narzędzi (jak `gcc`, `make`, `cmake`). To tutaj kod źródłowy transformowany jest do postaci binarnej.
2. **Stage Tester:** Dziedziczy po builderze i ma za zadanie uruchomić testy jednostkowe. Błąd na tym etapie wyzwala mechanizm fail-fast i natychmiastowo ubija cały proces CI.
3. **Stage Deploy:** Krytyczny krok końcowy, oparty przeważnie na najlżejszych obrazach (np. `alpine` lub `scratch`). 

Poniżej przedstawiono przykładowy kod implementujący tę strategię:
```dockerfile
# 1. BUILDER
FROM gcc:latest AS builder
WORKDIR /app
COPY src/ ./src/
COPY Makefile .
RUN make clean && make all

# 2. TESTER
FROM builder AS tester
RUN make test || exit 1

# 3. DEPLOY
FROM alpine:latest AS deploy
WORKDIR /app
# Ekstrakcja tylko samej biblioteki wygenerowanej w etapie 'builder'
COPY --from=builder /app/build/libapp.so .
# Instalacja absolutnego minimum bibliotek uruchomieniowych
RUN apk add --no-cache libstdc++ 
```
Przenoszone są jedynie skompilowane pliki wynikowe. Dzięki usunięciu kompilatorów, nagłówków czy skryptów źródłowych, zjawisko podatności na ataki zostaje zminimalizowane, a sam obraz "chudnie" z setek megabajtów do absolutnego minimum.

# 4. Zarządzanie artefaktami
Pomyślne zakończenie działania potoku CI/CD zawsze powinno skutkować powstaniem artefaktu. Jest to ściśle zwersjonowana i gotowa do użycia paczka (np. archiwum, biblioteka `.so`) bądź obraz kontenera.

## Ekstrakcja binarek i archiwizacja
Kiedy produktem końcowym jest biblioteka współdzielona, praktykuje się tworzenie nieuruchomionego kontenera tymczasowego. Służy on wyłącznie za nośnik pozwalający skopiować skompilowany plik na maszynę agenta Jenkinsa, skąd następnie pakuje się go do paczki dystrybucyjnej.

W pliku `Jenkinsfile` realizuje się to za pośrednictwem następującego skryptu:
```bash
# Wygenerowanie kontenera bez jego faktycznego uruchamiania
CONTAINER_ID=$(docker create ${IMAGE_NAME}:${APP_VERSION})

# Przeniesienie pliku z wnętrza kontenera do obszaru roboczego
docker cp ${CONTAINER_ID}:/app/libapp.so ./libapp.so

# Utylizacja tymczasowego kontenera
docker rm -v ${CONTAINER_ID}

# Kompresja pliku do formatu archiwum tar.gz
tar -czvf my-app-lib-${APP_VERSION}.tar.gz libapp.so
```
W dalszej kolejności Jenkins robi użytek ze swojej natywnej dyrektywy `archiveArtifacts`, która na stałe przywiązuje wygenerowane pliki do historii konkretnego uruchomienia w panelu:
```groovy
stage('Archive Artifacts') {
    steps {
        archiveArtifacts artifacts: '*.tar.gz', fingerprint: true
    }
}
```

## Dynamiczne wersjonowanie
Opisana technika gwarantuje, że każde przejście pipeline'u automatycznie znakuje pliki w oparciu o unikalną zmienną środowiskową `${env.BUILD_NUMBER}`. Skutkuje to bezbłędną identyfikowalnością logistyczną. Każdy artefakt zyskuje swój własny, niepowtarzalny stempel wersji powiązany z odpowiednim commitem i przebiegiem procesu CI, co bywa zbawienne przy późniejszym debugowaniu na środowisku produkcyjnym.

# 5. Lista kontrolna (Definition of Done)
Aby uznać proces CI/CD za wiarygodny, musi charakteryzować się on pełnym determinizmem. Oznacza to, że każde jego wywołanie z tymi samymi parametrami wejściowymi powinno dać identyczny, dający się przewidzieć wynik.

## Higiena środowiska i blokada pamięci podręcznej (cache)
By wyeliminować błędy spowodowane obecnością "starych" plików w systemie (np. zasobów z poprzednich iteracji), stosuje się dwie fundamentalne reguły:
- **Clean Workspace:** Użycie funkcji `cleanWs()` daje nam pewność, że na koniec każdego przebiegu folder roboczy zostanie bezwzględnie wyczyszczony.
  ```groovy
  post {
      always {
          cleanWs()
          sh 'docker system prune -f'
      }
  }
  ```
- **No-cache Build:** Aplikowanie flagi `--no-cache` w momencie tworzenia obrazów Dockera zmusza system do odrzucenia starych warstw pamięci podręcznej. Dzięki temu mamy gwarancję, że pobierane są tylko najnowsze zależności, a cały kod ulega prawidłowej rekompilacji.

## Weryfikacja kryteriów akceptacji
Nim dany artefakt zostanie zatwierdzony jako zdatny do wdrożenia, potok Jenkinsa musi potwierdzić zestaw twardych wymogów (tzw. Definition of Done):
1. **Kompilacja:** Przejście etapu Builder bez błędów wyjścia (oczekiwany kod błędu: `exit code 0`).
2. **Automatyzacja testów:** Zaliczenie wszystkich testów jednostkowych (np. napisanych w Google Test) w dedykowanym środowisku Testera.
3. **Audyt rozmiaru i bezpieczeństwa:** Potwierdzenie, że ostateczny obraz składa się z okrojonego systemu bazowego, pozbawionego powłoki bash czy zbędnych menedżerów pakietów.
4. **Smoke Testing:** Przeprowadzenie błyskawicznego testu wdrożeniowego (sanity check), weryfikującego fizyczną obecność skompilowanego pliku oraz jego powiązania biblioteczne.
   ```bash
   # Kontrola obecności wygenerowanej biblioteki w obrazie końcowym
   docker run --rm ${IMAGE_NAME}:${APP_VERSION} ls -la /app/libapp.so
   
   # Sprawdzenie zależności pod kątem brakujących powiązań (ldd)
   docker run --rm ${IMAGE_NAME}:${APP_VERSION} ldd /app/libapp.so
   ```
5. **Śledzenie pochodzenia (Identyfikowalność):** Obliczenie i zapisanie sum kontrolnych (MD5/SHA256) wygenerowanego artefaktu oraz poprawne otagowanie go unikalnym numerem buildu.

# 6. Wnioski
Rozbudowane potoki CI/CD bezpowrotnie przenoszą nas od zawodnych, ręcznych procedur wdrażania oprogramowania do powtarzalnej, skodyfikowanej automatyzacji. Synteza deklaratywnych rozwiązań Jenkins Pipeline z wieloetapowym budowaniem (Multi-stage Docker Builds) owocuje stworzeniem środowiska niezwykle bezpiecznego, szybkiego i transparentnego. 

Największą zaletą zaproponowanej architektury jest całkowite odizolowanie kolejnych zadań w dedykowanych kontenerach. Droga od bezstanowej kompilacji, przez zaawansowaną weryfikację jakości testami, aż do ostatecznego spakowania zwersjonowanego archiwum, odbywa się w warunkach ściśle kontrolowanych. Zastosowanie żelaznych reguł Definition of Done oraz konsekwentna higiena przestrzeni roboczej (`cleanWs()`, `--no-cache`) sprawiają, że jakakolwiek modyfikacja w repozytorium przechodzi przez bezlitosne sito automatycznej weryfikacji. W efekcie ryzyko wprowadzenia krytycznych błędów z winy czynnika ludzkiego zostaje zredukowane do absolutnego minimum, a stabilność dostarczanych rozwiązań drastycznie rośnie.
