# 1. Kontrola wersji i organizacja pracy
Git to fundament współczesnego wytwarzania oprogramowania, umożliwiający śledzenie zmian i bezpieczną pracę zespołową.

## Szyfrowanie połączenia
Tradycyjne hasła są wypierane przez bezpieczniejsze metody uwierzytelniania. Podstawą jest asymetryczna kryptografia, np. klucze SSH w wydajnym standardzie `ED25519`. W przypadku komunikacji po HTTPS stosuje się tokeny dostępowe (PAT) o ograniczonych uprawnieniach. Obowiązkowym standardem zabezpieczenia konta deweloperskiego jest również uwierzytelnianie dwuskładnikowe (2FA).

## Struktura środowiska i przepływ pracy
Praca dewelopera zazwyczaj opiera się na dwóch środowiskach: maszynie hosta (edytor kodu, GUI) oraz izolowanym systemie Linux (maszyna wirtualna/kontener). Pliki synchronizuje się na bieżąco, np. przez rozszerzenia typu *Remote-SSH*. 

Każda nowa funkcja powstaje na osobnej gałęzi, co zapobiega konfliktom:
```bash
$ git checkout -b BN419779
```
Integracja z główną kodbazą następuje poprzez *Pull Request i Code Review*. Dodatkowo, aby wymusić standardy (np. poprawny format opisów commitów), stosuje się lokalne skrypty automatyzujące, tzw. *Git Hooks* (w naszym przyadku wpływamy na git-msg).


# 2. Podstawy konteneryzacji
Docker to rynkowy standard, który pozwala na uruchamianie aplikacji w izolowanych środowiskach. W odróżnieniu od maszyn wirtualnych, kontenery są lekkie, ponieważ współdzielą jądro systemu 

## Środowisko i cykl życia kontenera

Silnik Dockera należy instalować z natywnych repozytoriów systemu Linux (omijając formaty takie jak Snap, które generują problemy z izolacją).

Kontenery bazują na gotowych obrazach (np. `ubuntu`, `alpine`), a ich cechą charakterystyczną jest efemeryczność. Kontener działa wyłącznie wtedy, gdy aktywny jest jego główny proces (PID 1). Aby pracować w nim interaktywnie, stosujemy polecenie:
```bash

$ docker run -it --rm busybox sh
```
Aby uniknąć zapchania dysku hosta przez zatrzymane instancje i stare obrazy, środowisko należy regularnie oczyszczać poleceniami `docker container prune` oraz `docker image prune -a`.

## Infrastructure as Code (Dockerfile)
Konfigurację środowiska zapisujemy w pliku Dockerfile. Dobrą praktyką jest minimalizacja warstw (instrukcji) oraz czyszczenie cache'u pobranych pakietów, co drastycznie zmniejsza wagę wynikowego obrazu:
```Dockerfile
FROM ubuntu:24.04
RUN apt-get update && apt-get install -y git && rm -rf /var/lib/apt/lists/*
WORKDIR /app
RUN git clone https://github.com/organizacja/repozytorium.git .
```
# 3. Powtarzalne budowanie i orkiestracja
Kontenery eliminują problem różnic środowiskowych ("u mnie działa"), zapewniając identyczne warunki kompilacji na każdej maszynie.

## Wieloetapowe budowanie
Proces tworzenia oprogramowania dzieli się na etapy zdefiniowane w plikach `Dockerfile`:  
1. Budowa (Build): Pobranie zależności i kompilacja kodu.
2. Testy: Uruchomienie testów na dokładnie tym samym obrazie, co gwarantuje wiarygodność wyników.
Aby chronić aplikację produkcyjną przed atakami i zmniejszyć jej rozmiar, do ostatecznego obrazu kopiuje się wyłącznie gotowe pliki binarne, odrzucając ciężkie kompilatory.

## Docker Compose
Zarządzanie wieloma powiązanymi kontenerami z poziomu terminala jest niewydajne. Narzędzie *Docker Compose* pozwala zapisać konfigurację usług i relacji między nimi (np. kolejność uruchamiania) w deklaratywnym pliku `docker-compose.yml`:
```Dockerfile
version: '3.8'
services: 
  builder: 
    build: { context: ., dockerfile: Dockerfile.build }
  tester:
    build: { context: ., dockerfile: Dockerfile.test }
    depends_on: [builder]
```
Podniesienie całego środowiska wymaga wtedy tylko komendy: `$ docker compose up --build`.  
# 4. Sieci, woluminy
Ulotność kontenerów wymaga odpowiedniego zarządzania danymi (stanem) oraz konfiguracji komunikacji sieciowej.
## Utrwalanie danych
Aby zachować pliki po usunięciu kontenera, stosuje się woluminy lub bind mounts. Przykładowo, zamiast instalować Git w obrazie budującym, można użyć tymczasowego "kontenera pomocniczego" do pobrania repozytorium na wolumin. Następnie ten sam wolumin podpina się do kontenera z kompilatorem:
```Bash
$ docker run -v kod_in:/src -v kod_out:/build ubuntu-builder
```
Nowoczesną alternatywą podczas budowania obrazów jest flaga `RUN --mount`, która optymalizuje wykorzystanie pamięci podręcznej i współdzielenie plików.  
## Wirtualne sieci (SDN)

Domyślna sieć Dockera wymusza komunikację po zmiennych adresach IP. Stworzenie dedykowanej sieci wirtualnej (`docker network create moja-siec`) aktywuje wewnętrzny serwer DNS. Dzięki temu kontenery mogą komunikować się po swoich nazwach (co testowano narzędziem iperf3). Wystawienie usług poza Dockera realizuje się mapowaniem portów (np. -p 8080:8080).
## SSH w kontenerze

Uruchamianie serwera SSH (`sshd`) w kontenerze to antywzorzec. Łamie on zasadę jednego procesu na kontener, powiększa obraz i generuje ryzyka bezpieczeństwa. Do diagnostyki i interakcji z systemem powinno się używać natywnego polecenia `docker exec`.

# 5. Środowisko CI/CD: Jenkins DinD
Jenkins to potężny, rynkowy standard służący do automatyzacji procesów ciągłej integracji i ciągłego dostarczania (CI/CD). Pełni on rolę centralnego orkiestratora, który łączy w spójną całość narzędzia takie jak Git i Docker.

## Architektura Docker-in-Docker (DinD) i trwałość danych
Zamiast instalować Jenkinsa bezpośrednio na serwerze hosta, wdraża się go jako skonteneryzowaną usługę. Ponieważ Jenkins sam musi budować obrazy i uruchamiać kontenery dla poszczególnych procesów kompilacji, stosuje się architekturę **DinD**. Składa się ona z dwóch współpracujących komponentów:
1. **Kontrolera (Controller):** Głównego kontenera z interfejsem webowym, zarządzającego logiką i zadaniami.
2. **Demona Dockera:** Osobnego kontenera wykonawczego (uruchomionego w trybie `--privileged`), który udostępnia środowisko uruchomieniowe Dockera kontrolerowi.

Usługi te komunikują się bezpiecznie poprzez certyfikaty TLS w obrębie współdzielonej sieci wirtualnej (zdefiniowanej np. w `docker-compose.yml`). Co kluczowe, aby zapobiec utracie konfiguracji, wtyczek oraz historii zadań po restarcie systemu, dane Jenkinsa muszą być bezwzględnie zmapowane na zewnętrzny wolumin (np. `jenkins-data`).

## Pipeline as Code i automatyzacja potoków
Wdrożenie skonteneryzowanego Jenkinsa to krok w stronę pełnej automatyzacji zadań, które wcześniej programista musiał wykonywać ręcznie (jak opisano w rozdziałach 3 i 4). 

Nowoczesne środowiska CI opierają się na koncepcji **Pipeline as Code**, gdzie definicja całego procesu (budowa, testy, wdrożenie) jest zapisana w pliku tekstowym (np. `Jenkinsfile`) i trzymana w repozytorium razem z kodem aplikacji. Jenkins potrafi nasłuchiwać zdarzeń z GitHuba (np. nowy commit lub Pull Request), a następnie automatycznie uruchomić potok: pobrać kod, wyizolować środowisko budujące w kontenerach, uruchomić w nich testy, a w przypadku sukcesu – wygenerować ostateczny, czysty obraz aplikacji. Architektura DinD sprawia, że cały ten proces jest bezpieczny, skalowalny i w 100% powtarzalny.
# 6. Wnoski
Praktyczne wdrożenie narzędzi DevOps drastycznie usprawnia proces wytwarzania oprogramowania. Integracja systemu Git z Dockerem zapewnia hermetyczność i powtarzalność budowania aplikacji. Wykorzystanie Docker Compose pozwala na zdefiniowanie całej infrastruktury w formie kodu, co redukuje błędy konfiguracyjne. Połączenie wieloetapowego budowania, świadomego zarządzania woluminami oraz własnych sieci z wewnętrznym DNS-em tworzy stabilne, zoptymalizowane środowisko pracy. Zwieńczeniem tego ekosystemu jest wdrożenie serwera Jenkins w architekturze DinD, stanowiące punkt wyjścia do pełnej automatyzacji potoków CI/CD.
